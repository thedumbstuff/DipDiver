# M3 · Lean execution chassis

> **Goal.** A Qlib-produced portfolio decision can be executed by Lean in backtest, paper, and (eventually) live — and the three produce the same orders.

## Why Lean (recap of ADR-004)

Lean is QuantConnect's open-source algorithmic trading engine. The properties we want from M3:

- **Backtest ↔ live parity is first-class.** Strategy code runs identically in backtest, paper, and live — same event loop, same fills logic.
- **Real broker plugins.** Native Alpaca, Interactive Brokers, Tradier, OANDA, Binance. M5 (broker adapters) only needs to extend, not replace.
- **Battle-tested.** Lean runs QuantConnect's hosted platform handling real money. We are not the first user of the execution path.

Lean is the **only** code path that reaches a broker. Period.

## Integration design

```
    M1 (Qlib LightGBM)          M2-lite (Alpha158Plus + LLM factors)
            │                                  │
            └────────── pred.pkl ──────────────┘
                            │
                            ▼
                  SignalSnapshot CSV
                  (date, symbol, score)
                            │
                            ▼
                  Lean Alpha Model
                  (reads CSV daily, emits Insights)
                            │
                            ▼
                  Lean Portfolio Construction
                  (top-k by score, n_drop rotation,
                   matches M1 backtest_params)
                            │
                            ▼
                  Lean Execution + Brokerage
                  (paper / live)
```

### Key choice: signals are pre-computed offline, not generated inside Lean

Qlib runs **outside** Lean's Docker container. M1/M2 produce a `pred.pkl`; we convert it to a CSV; Lean reads the CSV.

**Why not run Qlib inside Lean's container?**
- Lean's `quantconnect/lean` Docker image is tightly version-controlled and tested. Adding `pyqlib` + `lightgbm` + `torch` inside it is a 1+ GB build risk per Lean release.
- Running Python "research" code inside the execution container couples two very different concerns. We saw what that produces from the rdagent debacle.
- Pre-computing signals offline matches how real quant shops do it: research outputs *signals* to a feature store; execution consumes the feature store. Lean = execution.

**Trade-off acknowledged:** to backtest with M1's signals over an 18-month window, we generate the full signal CSV up front. For live, we'd run M1 daily, append today's signals to the CSV, Lean picks up the new row at its next bar. The "daily refresh" loop is part of M6 (forward-eval harness), not M3.

## Prerequisites

- **Docker Desktop installed and running.** Lean's CLI shells out to Docker for every backtest. Windows: same Docker Desktop that ran rdagent earlier. WSL2 backend fine.
- **Python 3.12 + `dipdiver[brain,m3]` installed.** The `[m3]` extra adds `lean>=1.0` (the CLI). Backtests run in Docker, so the host Python only needs the CLI + our adapter.
- **M1 baseline locked.** M3 reads `dipdiver/brain/baselines/locked/<hash>.json` for backtest parameters (universe, dates, costs, topk/n_drop) and reads the recorder's `pred.pkl` for signals.

## Acceptance criteria (from ROADMAP)

1. **Parity test.** Lean backtest orders match Qlib backtest orders within tolerance. Same instrument, same direction, same size on each rebalance day. Price differences only from intra-bar fill model.
2. **Paper-account run** matches Lean backtest for the same 5 days within slippage budget.

The parity test is the *engineering* deliverable. The paper run is the *operational* deliverable.

## Lean Insight schema (for reference)

```python
Insight(
    symbol,            # Symbol
    period,            # timedelta — how long the prediction is valid
    type,              # InsightType.Price or InsightType.Volatility
    direction,         # InsightDirection.Up / Down / Flat
    magnitude=None,    # float — expected percent change (optional)
    confidence=None,   # float in [0, 1] (optional)
    sourceModel=None,
)
```

Our SignalSnapshot.score maps to:
- `direction` = Up if score > 0 else Down (with Flat threshold near 0)
- `magnitude` = abs(score)
- `period` = 1 day (matching our M1 daily rebalance)

Portfolio construction picks top-k by `magnitude` (matching `backtest_params.topk` from M1 config).

## Build sequence

| # | Step | Output | Acceptance |
|---|---|---|---|
| 1 | Install Lean CLI + verify Docker connection | `lean --version` prints; `lean init` succeeds | Doc'd in `m3_setup.sh` |
| 2 | Export M1 signals to CSV | `lean_projects/dipdiver_dow30_lightgbm/data/signals.csv` | Row count = test-window-days × universe-size, no NaN scores |
| 3 | Author Lean algorithm | `dipdiver/adapters/lean/algorithm.py` reads CSV, emits Insights | Lean backtest produces non-zero P&L |
| 4 | Run a Lean backtest | `lean backtest dipdiver_dow30_lightgbm` exits 0 | `backtests/<ts>/` has report.json |
| 5 | Compare Lean vs Qlib orders | `m3_verify.py` prints per-day order diff | Order match >95% across rebalance days |
| 6 | Connect Alpaca paper, run 5 days | Live order log matches forward sim | Slippage stays within `0.5 * close_cost` budget |

Steps 3–5 are where the hard engineering is. Steps 1–2 are mechanical.

## Why this milestone is small in code but big in care

The code volume for M3 is probably 300–500 lines (signals adapter, Lean algorithm, parity test). The *care* is in:

- Getting Lean's universe, calendar, fill model, and cost model to mirror M1's Qlib backtest assumptions exactly. Mismatches will silently destroy parity.
- Symbol mapping: Qlib uses lowercase `aapl`, Lean uses `Symbol("AAPL")`. One-to-one but the conversion can't have surprises.
- Rebalancing timing: M1 rebalances on close, Lean's default is on open. We need to either set Lean's `Resolution.Daily` + schedule on close, or shift M1's signals one day forward.
- Cost parity: M1 uses `open_cost=0.0001, close_cost=0.0002` (Alpaca-style). Lean's default Alpaca brokerage model has different per-share fees. We override.

## Failure modes (anticipated)

| Symptom | Likely cause | Fix |
|---|---|---|
| `lean: command not found` | `pip install -e ".[m3]"` not run | Run the install |
| `docker: not running` from CLI | Docker Desktop off / WSL backend asleep | Start Docker Desktop |
| Lean backtest exits 0 but holds nothing | Algorithm couldn't find the signals CSV inside the container | Check the mount: signals CSV must be under the project's `data/` |
| Parity test: instrument count mismatch | Symbol case / suffix mismatch (AAPL vs aapl, RELIANCE.NS vs RELIANCE) | Add explicit mapping in `signals.py:signals_from_qlib_pred` |
| Parity test: orders match but P&L differs by >5% | Cost or fill-price model mismatch | Override `SetBrokerageModel()` and `SetSlippageModel()` in algorithm.py |

## Step 6: Alpaca paper trading (live execution)

### Why Alpaca-py directly, not `lean live deploy`

ADR-004's "Lean is the only code path that reaches a broker" assumed Lean's live mode was viable without subscription cost. It isn't — `lean live deploy` requires a paid QuantConnect tier (Researcher seat + Live Node ≈ $28/month). We accept the trade-off:

- **Backtest execution**: Lean (battle-tested, structural backtest ↔ live parity)
- **Live execution**: `scripts/m3_live_alpaca.py` calling Alpaca's free paper API directly

Both consume the same `signals.csv`. Both apply identical TopkDropoutStrategy logic — the shared piece is `dipdiver/adapters/alpaca/strategy.py:compute_target_holdings`, which mirrors the rotate-only loop in `lean_projects/dipdiver_dow30_lightgbm/main.py`. **If you change one, change the other.**

ADR-004 amended: Lean owns backtest; Alpaca-direct owns live. Cross-path parity is empirical (5-day paper run vs Lean backtest over the same window), not structural.

### Setup (one-time)

1. **Sign up** at [app.alpaca.markets/signup](https://app.alpaca.markets/signup) — free, no credit card.
2. Go to [the paper dashboard](https://app.alpaca.markets/paper/dashboard/overview) → "Your API Keys" → generate paper key + secret.
3. Paste into `.env.m2`:
   ```
   ALPACA_API_KEY=PK...
   ALPACA_API_SECRET=...
   ```
4. Install the `[m3]` extras if not already (adds `alpaca-py`):
   ```
   pip install -e ".[m3]"
   ```

### Daily run

```bash
# Dry-run first — shows the planned order list without submitting
python scripts/m3_live_alpaca.py --m1-config dow30_lightgbm.yaml --dry-run --force

# Real paper run, during market hours
python scripts/m3_live_alpaca.py --m1-config dow30_lightgbm.yaml

# After hours / weekend: use --force to bypass the "market closed" guard
python scripts/m3_live_alpaca.py --m1-config dow30_lightgbm.yaml --force
```

Each invocation:
- Auto-loads `.env.m2`
- Connects to Alpaca paper, snapshots account + current positions
- Loads the **latest signal date** in `signals.csv`
- Computes target via TopkDropoutStrategy
- Submits market orders for adds (notional = equity / topk) and `close_position` for removes
- Writes one JSON record to `logs/m3_live/<universe>/<YYYY-MM-DD>.json`

Idempotent — re-running on the same day with the same signal won't double-trade.

### Expected first-day output

On the very first run:
- Current positions: 0
- adds: top-10 by score (cold start)
- removes: empty
- 10 market BUY orders submitted

On subsequent runs (with stable signals.csv):
- adds + removes each up to `n_drop` (default 3)

### Verifying the 5-day paper test

The ROADMAP acceptance is "live order log matches forward sim, slippage stays within 0.5 × close_cost budget". After running for 5 trading days:

1. Collect the 5 `logs/m3_live/<universe>/*.json` files
2. Run a Lean backtest covering the same 5-day window with the same signals
3. Diff the order lists day-by-day (a script — `scripts/m3_verify_live.py` — is future work; the structure mirrors `m3_verify.py` but reads JSON logs on both sides instead of Lean's order-events JSON)

### A note about signal freshness

`signals.csv` as generated today covers M1's test window (2024-01-01 → 2025-12-31). For real forward live trading you'd want a daily refresh that re-fetches recent OHLCV from Yahoo, re-runs M1's predict on today's date, and appends one new row per symbol to `signals.csv`. That refresh is intentionally **not** built yet — it's the natural M6 (forward-eval harness) deliverable. For the 5-day paper test, using the most-recent stale signal is fine to validate the **mechanics**.
